## Rationale

### Analysis of Changes (commit dccbafc)

The commit introduced a **conditional assignment sanitizer** for C taint tracking that reduces false positives when tainted size variables are bounds-checked before reaching buffer overflow or memory corruption sinks.

**Code changes:**
1. **`taint_tracker.py`** added ~370 lines implementing:
   - `TaintState.add_sanitization()` / `is_sanitized_for()` methods
   - `_handle_if_sanitizer()` - detects `if (n > max) n = max;` pattern
   - `_check_ternary_clamp()` - detects `(n > max) ? max : n` pattern
   - Category-based sanitization: `_SIZE_SANITIZABLE_CATEGORIES` = {CWE-119, CWE-120, CWE-190}
   - Sanitized sinks emit `sanitizer="conditional_bounds_check"` in TaintPath

2. **`test_taint_c_paths.py`** added comprehensive test class `TestConditionalSanitizer` (15 test methods)

3. **`conditional_bounds.c`** added safe sample fixture demonstrating 9 clamp variants

**Documentation coverage:**

`CLAUDE.md` line 156 was updated in the same commit (dccbafc) to document this feature as Known Limitation #10. The limitation entry:
- Clearly states what IS recognized (if-statement clamp, ternary clamp)
- Clearly states what is NOT recognized (macros, early-return guards, bitwise masks)
- Explains the confidence impact (downgrade from "confirmed" to "likely")
- Lists the affected sink categories (CWE-119/120/190)

This is **accurate and complete** documentation for the feature. The limitation is correctly positioned as a "partial coverage" scenario rather than a missing feature, which aligns with the implementation.

`README.md` was not modified in commit dccbafc because the conditional sanitizer does not introduce:
- New CLI flags or commands
- New environment variables
- New output fields (sanitizer field already existed in TaintPath schema)
- Changes to supported languages or detection rate estimates

The general Known Limitations section in README.md already covers the intraprocedural constraint and C-specific limitations (no preprocessor, no struct tracking), which provide sufficient context for end users.

### Verdict Justification

Both documentation files are **current** relative to the codebase state after commit dccbafc. The feature is appropriately documented in `CLAUDE.md` (developer guide) with sufficient technical detail, and `README.md` (user guide) correctly omits internal implementation details that do not affect the user-facing API or behavior.
