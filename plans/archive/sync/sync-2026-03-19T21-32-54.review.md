## Rationale

### Why these changes are needed

**Commit 992e1e5** introduced a major architectural enhancement: Semgrep is now the primary scanner backend with tree-sitter as a fallback. This fundamentally changes how the Hunter phase works:

1. **README.md is outdated**: It still describes the system as "Uses tree-sitter" (line 4) when in fact the default is now Semgrep (when available). This is misleading to new users.

2. **Missing environment variable documentation**: Three new environment variables (`DCS_SCANNER_BACKEND`, `DCS_SEMGREP_TIMEOUT`, `DCS_SEMGREP_RULES_PATH`) are documented in CLAUDE.md but missing from README.md's environment variable table.

3. **Limitation language is imprecise**: The "Query brittleness" limitation (README.md line 246, CLAUDE.md line 152) is now backend-specific. The current wording implies all scans suffer from this limitation, when in fact Semgrep scans do NOT have this limitation.

4. **Architecture diagrams should reflect reality**: Both README.md line 45 and CLAUDE.md line 22 describe the Hunter phase as "tree-sitter parse" when the actual implementation is now backend-agnostic.

### Evidence from commit 992e1e5

- Added `src/deep_code_security/hunter/scanner_backend.py` - backend selection logic
- Added `src/deep_code_security/hunter/semgrep_backend.py` - 599 lines of Semgrep integration
- Added `src/deep_code_security/hunter/treesitter_backend.py` - refactored existing logic
- Added 14 Semgrep YAML rule files in `registries/semgrep/{python,go,c}/`
- Added `DCS_SCANNER_BACKEND`, `DCS_SEMGREP_TIMEOUT`, `DCS_SEMGREP_RULES_PATH` to config.py (lines 56-66)
- Updated CLI to show scanner backend in `dcs status` output (cli.py lines 440-455)
- CLAUDE.md already updated with scanner backend design decision (line 83) and env vars (lines 107-109)

### Impact of not updating

- **User confusion**: New users will think tree-sitter is the only/primary scanner, missing out on Semgrep's superior pattern matching
- **Documentation drift**: README.md and CLAUDE.md will continue to diverge
- **Incomplete configuration guidance**: Users won't know they can tune Semgrep timeouts or rules paths
- **Misleading limitation statements**: Users may avoid the tool thinking all scans have query brittleness, when Semgrep scans do not
