# Plan: Textual TUI Frontend (`dcs tui`)

## Status: APPROVED

## Goals

1. **Provide a terminal-native interactive frontend** for deep-code-security using the Textual framework, giving developers a keyboard-driven workflow for target selection, scan configuration, progress monitoring, and result browsing.
2. **Implement real-time progress streaming** from long-running `dcs` CLI commands (scans of large C projects like OpenSSL can take 45+ minutes), displaying live stderr output in a scrollable log panel.
3. **Implement a consistent report storage layout** with a new `DCS_OUTPUT_DIR` env var (default `~/.dcs/reports/`), project-name subdirectories, and ISO-timestamp run directories, each containing `meta.json` plus the report files generated by existing formatters.
4. **Implement a run history browser** that reads `meta.json` files from the report storage layout, showing past scans per project with sortable columns (date, scan type, findings count, duration).
5. **Add the `dcs tui` CLI entrypoint** as a new Click command that launches the Textual app.

## Non-Goals

- **Replacing the existing CLI.** The TUI wraps the CLI via subprocess; all existing commands (`dcs hunt`, `dcs full-scan`, `dcs fuzz`, `dcs hunt-fuzz`, etc.) remain unchanged and usable independently.
- **SARIF report viewing.** Users view SARIF files in VS Code (SARIF Viewer extension) or GitHub Code Scanning. The TUI provides a button to open the file in the system default application but does not render SARIF inline.
- **HTML report rendering inside the terminal.** HTML reports are generated by the existing HTML formatter and stored in the report directory. The TUI provides a button to open them in the default browser but does not render HTML in the terminal.
- **Importing or calling hunter/auditor/fuzzer Python APIs directly.** The TUI communicates exclusively through `dcs` CLI subprocess invocations for scanning. The only exception is post-scan format conversion, where the TUI calls `shared.formatters` directly to avoid re-running the entire scan pipeline (see Deviation D-3 in Context Alignment).
- **New MCP tools.** The TUI is a local developer tool, not an MCP surface.
- **Fuzzer consent management.** The TUI does not auto-consent to API transmission. Users must pass `--consent` or set `DCS_FUZZ_CONSENT=true` before running fuzz-dependent scans.
- **Editing source code or applying patches.** The TUI is read-only with respect to the scanned codebase.
- **Custom Textual CSS theming.** The TUI uses Textual's built-in dark theme. Custom theming is a future enhancement.
- **Free-form CLI argument pass-through.** All scan options are exposed as explicit, validated UI controls. Users who need custom flags use the `dcs` CLI directly.

## Assumptions

1. `textual>=0.70.0` is available as an optional dependency (`pip install deep-code-security[tui]`). The TUI does not load at import time; it is imported lazily by the `dcs tui` Click command.
2. The existing `dcs` CLI commands emit structured progress information on stderr (e.g., `[1/3] Scanning...`, `Found N findings in M files`) that the TUI can parse and display.
3. The `dcs hunt` and `dcs full-scan` commands support `--format json` with `--output-file` for writing machine-readable results to a specific path.
4. The terminal supports 256 colors and Unicode (standard for modern terminals on macOS, Linux, and Windows Terminal).
5. `asyncio.create_subprocess_exec()` is available and functional on all target platforms (Python 3.11+).
6. The user's home directory (`~/.dcs/`) is writable. If not, the TUI falls back to a temp directory with a warning.

## Proposed Design

### Architecture Overview

```
src/deep_code_security/tui/
    __init__.py           # __all__ exports, lazy check for textual availability
    app.py                # DCSApp(App) -- main Textual application
    models.py             # RunMeta, RunHistoryEntry, ScanConfig Pydantic models
    runner.py             # ScanRunner -- async subprocess wrapper, progress parsing
    storage.py            # ReportStorage -- DCS_OUTPUT_DIR layout, meta.json I/O
    screens/
        __init__.py
        target_select.py  # TargetSelectScreen -- filesystem browser + path entry
        scan_config.py    # ScanConfigScreen -- scan type, options, language filter
        scan_progress.py  # ScanProgressScreen -- live log, progress bar, cancel
        results_view.py   # ResultsViewScreen -- findings summary from meta.json
        history.py        # HistoryScreen -- per-project run history browser
```

The TUI module is entirely self-contained in `src/deep_code_security/tui/`. It does NOT import from `hunter`, `auditor`, `architect`, `fuzzer`, `bridge`, or `mcp` modules. The imports from the main package are:

- `deep_code_security.shared.config.Config` -- to read `DCS_ALLOWED_PATHS` for path validation feedback (display only; the actual validation happens in the subprocess).
- `deep_code_security.__version__` -- for the title bar.
- `deep_code_security.shared.formatters` -- to convert JSON scan output to SARIF/HTML formats without re-running the scan (see Deviation D-3).

All analysis is performed by spawning `dcs hunt`, `dcs full-scan`, `dcs fuzz`, or `dcs hunt-fuzz` as async subprocesses. Format conversion (JSON to SARIF/HTML) is performed in-process using the existing formatter registry.

### Component 1: Report Storage (`storage.py`)

#### Layout

```
~/.dcs/reports/
    {project-name}/
        2026-03-20-14-30-00-483927/
            meta.json
            hunt.sarif
            hunt.html
            hunt.json
        2026-03-20-15-00-00-192847/
            meta.json
            full.sarif
            full.html
            full.json
        2026-03-21-10-00-00-738291/
            meta.json
            hunt-fuzz.sarif
            hunt-fuzz.json
```

The `{project-name}` is derived from the target path's basename. For example, scanning `/Users/imurphy/projects/openssl` produces `project-name = openssl`. If the target is a single file, the project name is the file's parent directory basename.

The timestamp directory uses the format `YYYY-MM-DD-HH-MM-SS-ffffff` (UTC, microsecond precision, no colons for filesystem compatibility). Microsecond precision prevents directory name collisions when two scans of the same project start within the same second.

#### `DCS_OUTPUT_DIR` Environment Variable

| Variable | Default | Description |
|----------|---------|-------------|
| `DCS_OUTPUT_DIR` | `~/.dcs/reports/` | Root directory for TUI report storage. Expanded via `Path.expanduser()`. |

The `DCS_OUTPUT_DIR` is read directly by `ReportStorage.__init__()` in `tui/storage.py`. It is NOT added to `shared/config.py` -- it is a TUI-only concern (see Deviation D-5 in Context Alignment).

        ...

    def get_output_dir(self) -> Path:
        """Return the resolved output directory path."""
        ...

### Component 2: Scan Runner (`runner.py`)

The `ScanRunner` wraps `asyncio.create_subprocess_exec()` to launch `dcs` CLI commands and stream their output in real time.

```python
class ScanRunner:
    """Async subprocess runner for dcs CLI commands."""

    def __init__(
        self,
        scan_config: ScanConfig,
        run_dir: Path,
        on_stderr_line: Callable[[str], None] | None = None,
        on_complete: Callable[[int], None] | None = None,
    ) -> None:
        """Initialize the runner.

        Args:
            scan_config: Configuration for the scan to run.
            run_dir: Directory to write report files to.
            on_stderr_line: Callback invoked for each line of stderr output
                            (progress updates, phase transitions).
            on_complete: Callback invoked when the subprocess exits.
        """
        ...

    def build_command(self) -> list[str]:
        """Build the dcs CLI command as a list of arguments.

        Returns a list suitable for asyncio.create_subprocess_exec().
        Never uses shell=True. All scan options are derived from
        explicitly-typed ScanConfig fields -- no free-form arguments.
        """
        ...

    async def run(self) -> RunMeta:
        """Execute the scan and return the run metadata.

        Launches the subprocess, streams stderr to the callback,
        captures stdout (report output), and writes report files
        to run_dir.

        The scan runs ONCE with --format json. After completion,
        SARIF and HTML are derived from the JSON output using the
        shared formatter registry (in-process, no additional subprocess).

        Returns a populated RunMeta.
        """
        ...

    async def cancel(self) -> None:
        """Cancel the running scan by sending SIGTERM to the subprocess.

        Waits up to 5 seconds for graceful termination, then sends SIGKILL.
        """
        ...
```

**Key design decisions:**

1. **`sys.executable` for subprocess launch.** The runner uses `sys.executable` to locate the Python interpreter, then runs `[sys.executable, "-m", "deep_code_security.cli", "hunt", ...]` rather than looking up `dcs` on `$PATH`. This guarantees the same Python environment is used, avoiding issues where `dcs` might resolve to a different installation.

   However, the exact invocation is `asyncio.create_subprocess_exec(sys.executable, "-m", "deep_code_security.cli", ...)` -- NOT `subprocess.run(..., shell=True)`.

2. **Single scan invocation + in-process format conversion.** The runner invokes the `dcs` command exactly ONCE with `--format json --output-file {run_dir}/{prefix}.json` to capture the machine-readable results. After the subprocess completes, the runner reads the JSON output file and uses the `deep_code_security.shared.formatters` registry directly to produce SARIF and HTML files in-process. Specifically, it calls `get_formatter("sarif")` and `get_formatter("html")`, then invokes `format_hunt()`, `format_full_scan()`, `format_fuzz()`, or `format_hunt_fuzz()` as appropriate.

   This is an explicit, justified exception to the "TUI wraps CLI" constraint (see Deviation D-3 in Context Alignment). The formatters are pure data transformers in `shared/formatters/` -- they take structured data in and produce string output. They do not invoke hunter, auditor, fuzzer, or any analysis pipeline. Calling them directly avoids a 200% time penalty from re-running the full scan pipeline for each output format.

   The `prefix` matches the scan type (`hunt`, `full`, `hunt-fuzz`). If SARIF or HTML generation fails (e.g., the formatter does not support fuzz output, or the JSON is malformed), the runner logs a warning and continues -- the JSON report is always the primary artifact.

3. **No free-form `extra_args`.** The `ScanConfig` model exposes only explicitly-typed fields (scan_type, languages, severity_threshold, skip_verify, ignore_suppressions). There is no `extra_args: list[str]` field. This eliminates the risk of users injecting conflicting flags (`--output-file`, `--format`) that would break TUI operation. Users who need custom flags use the `dcs` CLI directly.

4. **Stderr parsing for progress.** The runner parses known patterns from stderr:
   - `Scanning {path}...` -- scan started
   - `[1/3] Scanning...`, `[2/3] Verifying...`, `[3/3] Generating...` -- phase transitions
   - `Found {N} findings in {M} files` -- intermediate results
   - `Error:` -- error detection

   These patterns are extracted into constants in `runner.py` for maintainability. Unrecognized stderr lines are passed through verbatim to the log panel.

5. **`findings_count` extraction.** After the JSON report is written, the runner reads it and extracts the findings count for `meta.json`. The extraction paths per scan type are:
   - **hunt**: `output["total_count"]`
   - **full-scan**: `output["total_count"]`
   - **hunt-fuzz**: `output["hunt_result"]["total_count"]`
   - **fuzz**: `output["summary"]["unique_crash_count"]`

   If JSON parsing fails (malformed output, truncated file, disk full), the runner falls back to the defaults (`findings_count=0`, `backend_used="unknown"`) and logs a warning.

6. **`backend_used` extraction.** The extraction paths per scan type are:
   - **hunt**: `output["stats"]["scanner_backend"]`
   - **full-scan**: `output["hunt_stats"]["scanner_backend"]`
   - **hunt-fuzz**: `output["hunt_result"]["stats"]["scanner_backend"]`
   - **fuzz**: not applicable (defaults to `"unknown"`)

   If the key is not present, falls back to `"unknown"`.

#### Platform File Opener

The [Open] buttons on `ResultsViewScreen` and `HistoryScreen` use platform-specific methods to open report files:

- **macOS** (`sys.platform == "darwin"`): `asyncio.create_subprocess_exec("open", file_path)`
- **Linux** (`sys.platform == "linux"`): `asyncio.create_subprocess_exec("xdg-open", file_path)`
- **Windows** (`sys.platform == "win32"`): `os.startfile(file_path)` (stdlib function, no subprocess needed, no `shell=True` concern)

The `os.startfile()` approach on Windows is preferred because `start` is a `cmd.exe` builtin, not a standalone executable, and using it with `create_subprocess_exec` would require `shell=True` which is prohibited. `os.startfile()` is a stdlib function that calls the Windows `ShellExecute` API directly.

All file paths passed to the opener are constructed from `DCS_OUTPUT_DIR` + sanitized project name + timestamp directory + known filename. These paths are fully deterministic and not user-controlled at the point of invocation.

### Component 4: Screens

#### `TargetSelectScreen`

Uses Textual's `DirectoryTree` widget for filesystem browsing. Starts at the current working directory. Users can also type a path directly into an `Input` widget. The screen validates that the path exists and is a directory (or file for single-file scans) before enabling the [Select] button.

The screen reads `DCS_ALLOWED_PATHS` from the config and displays a note if the selected path is outside the allowed paths (informational -- the actual validation happens when the `dcs` subprocess runs).

#### `ScanConfigScreen`

Presents scan options via Textual widgets:

- `RadioSet` for scan type (`hunt`, `full-scan`, `hunt-fuzz`, `fuzz`)
- `SelectionList` for language filter (checkboxes for Python, Go, C)
- `Select` dropdown for severity threshold
- `Switch` toggles for `--skip-verify` and `--ignore-suppressions`

All scan options are explicitly enumerated UI controls. There is no free-form text input for additional CLI arguments. Users who need custom flags use the `dcs` CLI directly.

The scan type selection dynamically shows/hides options that only apply to certain scan types (e.g., `--skip-verify` only for `full-scan`; `--consent` only for `fuzz` and `hunt-fuzz`).

#### `ScanProgressScreen`

This is the most critical screen for the 45-minute OpenSSL scan use case. It provides:

1. **Phase indicator** -- A `Static` widget showing the current phase (e.g., `[2/3] Verifying 47 findings...`). Updated by parsing stderr output.
2. **Elapsed time** -- A `Static` widget updated every second by a Textual timer, showing `HH:MM:SS` elapsed.
3. **Scrollable log** -- A `RichLog` widget that appends each line of stderr output in real time. Auto-scrolls to bottom but allows manual scroll-up to review earlier output.
4. **Cancel button** -- Sends `SIGTERM` to the subprocess. If the process does not exit within 5 seconds, sends `SIGKILL`. Updates the progress display to show "Cancelling..." then "Cancelled."

The screen uses Textual's `work` decorator to run the `ScanRunner.run()` coroutine in a worker thread, posting messages back to the UI thread for display updates.

#### `ResultsViewScreen`

Displays a summary from the completed `RunMeta`:

- Target path, scan type, duration, findings count, backend used
- List of generated report files with [Open] buttons
- The [Open] button uses the platform file opener described above
- [New Scan] and [History] navigation buttons

#### `HistoryScreen`

Displays a `DataTable` of past runs for a selected project:

| Column | Description |
|--------|-------------|
| Date | ISO timestamp from meta.json |
| Scan Type | hunt / full-scan / hunt-fuzz / fuzz |
| Findings | findings_count from meta.json |
| Duration | formatted as HH:MM:SS |
| Backend | scanner backend used |
| Exit Code | 0 (green) or non-zero (red) |

Sorted by date descending (most recent first). A `Select` dropdown at the top switches between projects. The [View] button on a selected row opens the run directory in the platform file manager.

The `dcs tui` command is added to `cli.py`:

def tui() -> None:

## Risks

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| Textual API instability (pre-1.0) | Medium | Medium | Pin to `>=0.70.0` to get stable widget APIs (DirectoryTree, DataTable, RichLog). Textual has been rapidly stabilizing and the widgets we use are mature. Monitor changelogs on version bumps. |
| Terminal compatibility issues (Windows, SSH, tmux) | Low | Low | Textual handles terminal compatibility internally. The TUI is primarily a developer convenience; the CLI remains the primary interface. |
| `meta.json` corruption from concurrent TUI instances scanning the same project | Low | Low | Each run gets a unique microsecond-precision timestamp directory. Two concurrent scans of the same project produce two distinct run directories. No locking required. |
| Report storage grows unbounded | Medium | Low | Document that users should periodically clean `~/.dcs/reports/`. A future `dcs prune` command could automate this. The TUI itself does not delete reports. |
| `sys.executable` may not resolve to a working `dcs` installation | Low | Medium | The TUI checks that `dcs --version` succeeds during startup and shows a diagnostic error if it fails. |
| Subprocess stderr parsing breaks if CLI output format changes | Medium | Low | Stderr patterns are extracted into named constants in `runner.py`. Tests verify pattern matching against sample output. Changes to CLI stderr are visible in the same repository. |
| In-process SARIF/HTML conversion fails on malformed JSON | Low | Low | Format conversion is wrapped in try-except. On failure, the JSON report (always present) is the primary artifact; SARIF/HTML are best-effort. A warning is logged and the user is shown only the available report files. |

## Trust Boundary Analysis

### TUI Module

The TUI sits entirely on the user's side of the trust boundary. It is a local developer tool that:

- **Reads the filesystem** to display the directory tree and read `meta.json` files. All filesystem reads are confined to the `DCS_OUTPUT_DIR` directory tree (for history) and the user's working directory (for target selection). No file content is executed or evaluated.
- **Spawns `dcs` CLI subprocesses** using `asyncio.create_subprocess_exec()` with list-form arguments. The subprocesses inherit the user's environment, including `DCS_ALLOWED_PATHS`. Path validation is performed by the subprocess (via `path_validator.py`), not by the TUI.
- **Writes to `DCS_OUTPUT_DIR`** to store `meta.json` and report files. Write paths are constructed programmatically from the project name (sanitized: stripped to `[a-zA-Z0-9._-]`, no `..`, no `/`, max 64 chars) and microsecond-precision timestamp. No user-controlled path components beyond the target path basename are used.
- **Imports `shared.formatters`** to convert JSON scan output to SARIF/HTML formats in-process. The formatters are pure data transformers that take structured data in and produce string output. They do not invoke any analysis pipeline, access the filesystem (beyond the report files being written), or execute code. This import is the single exception to the "TUI wraps CLI only" constraint and is justified by the 200% time penalty of re-running the full scan pipeline for each format.
- **Opens files using platform-native methods.** macOS: `["open", path]`. Linux: `["xdg-open", path]`. Windows: `os.startfile(path)`. The file paths are constructed from `DCS_OUTPUT_DIR` + project name + timestamp + known filename. These paths are fully deterministic and not user-controlled at the point of invocation. No `shell=True` is used on any platform.

**Conclusion:** The TUI introduces no new trust boundaries. All security-sensitive operations (path validation, scanning, sandbox execution) are delegated to the existing `dcs` CLI subprocess, which enforces `DCS_ALLOWED_PATHS`, `PathValidator`, and container security policy independently.

### Report Storage

The `meta.json` file is written by the TUI and read by the TUI. It is not processed by any other component. The file is loaded with `json.loads()` (not `yaml.load()`) and validated through a Pydantic model. No code execution is involved.

The report files (`.sarif`, `.html`, `.json`) are generated by the existing `dcs` CLI formatters (JSON via subprocess, SARIF/HTML via in-process conversion). The TUI opens them for viewing but does not parse or execute their contents.

No arguments or options. No input validation required.

| Input | Validation |
|-------|-----------|

| Input | Validation |
|-------|-----------|
| `meta.json` content | Parsed with `json.loads()`. Validated through `RunMeta` Pydantic model. Invalid files are skipped with a warning log (not displayed to user). |

## Known Limitations

1. **Stderr parsing is pattern-based.** If CLI stderr output format changes, the TUI's phase indicator and progress detection may stop working. The patterns are extracted into constants for easy maintenance.
2. **No report pruning.** The TUI does not delete old reports. Users must manually clean `~/.dcs/reports/` or use filesystem tools.
3. **No inline SARIF or HTML viewing.** SARIF and HTML reports are opened in external applications. Terminal-based viewing is not provided.
4. **Fuzz consent not handled by TUI.** Users must configure `DCS_FUZZ_CONSENT=true` or click through the consent prompt in the subprocess for fuzz-dependent scans.
5. **Single-file target scanning.** When the target is a file rather than a directory, the project name is derived from the parent directory, which may be unexpected for users scanning individual files.
6. **No Windows testing.** Textual supports Windows Terminal, but the TUI has not been tested on Windows. The `os.startfile()` / `open` / `xdg-open` detection via `sys.platform` is best-effort.
7. **Project name collisions.** Scanning two directories with the same basename (e.g., `/home/user/a/openssl` and `/opt/vendor/openssl`) produces the same project name. Their run histories are silently merged in the history view. Individual runs are distinguishable by `target_path` in `meta.json`.

### Deviations from Established Patterns

**D-1: TUI does not import internal analysis modules.** Unlike the CLI (`cli.py`) which imports `HunterOrchestrator`, `AuditorOrchestrator`, etc. directly, the TUI wraps the CLI via subprocess for all scanning operations. This is an intentional deviation to avoid coupling the TUI to internal APIs. Justification: the TUI is a presentation layer that should survive internal API refactoring. Subprocess invocation also makes it language-agnostic if the CLI is ever rewritten.

**D-2: TUI tests have their own `make test-tui` target.** Unlike other modules whose tests are included in `make test`, TUI screen/app tests are separate because they require the optional `textual` dependency. This follows the same pattern as `make test-integration` which requires Docker/Podman. Note: tests for `models.py`, `storage.py`, and `runner.py` are included in `make test` because these modules do not depend on `textual`.

**D-3: TUI calls `shared.formatters` directly for format conversion.** The plan's non-goal states the TUI "communicates exclusively through `dcs` CLI subprocess invocations." The exception is post-scan format conversion: after the single JSON-format scan completes, the runner uses `shared.formatters.get_formatter("sarif")` and `get_formatter("html")` to produce SARIF and HTML files in-process rather than re-running the full scan pipeline via subprocess. This is justified because:
- The formatters are pure data transformers in `shared/formatters/`. They take structured data in and produce string output. They do not invoke hunter, auditor, fuzzer, or any analysis pipeline.
- Without this, the TUI would need to run the full scan pipeline 3 times (JSON, SARIF, HTML), adding a 200% time penalty. For a 45-minute OpenSSL scan, this means 135 minutes instead of 45.
- The `shared/formatters/` module is a stable, tested, public API with a registry pattern (`get_formatter()`). It is not an internal analysis API.

**D-4: `DCS_OUTPUT_DIR` write paths use direct `Path.mkdir()` rather than `path_validator.py`.** The `path_validator.py` module enforces the `DCS_ALLOWED_PATHS` MCP trust boundary -- it prevents MCP tools from accessing arbitrary files on behalf of a remote client. The TUI is a local tool with no external trust boundary; write paths are programmatically constructed from a sanitized project name derived from the target basename (stripped to `[a-zA-Z0-9._-]`, no `..`, no `/`, max 64 chars) and a generated timestamp, not from user-typed free-form input. Routing these through `path_validator.py` would inappropriately apply MCP access controls to a local developer tool.

**D-5: `DCS_OUTPUT_DIR` is NOT added to `shared/config.py`.** The plan originally proposed adding `output_dir` to the shared `Config` class. This has been revised: `DCS_OUTPUT_DIR` is read directly by `tui/storage.py` via `os.environ.get()`. This prevents false expectations that other components (CLI, MCP server) respect this variable, and avoids bleeding TUI-only configuration into the shared config singleton. The existing `DCS_FUZZ_OUTPUT_DIR` (for fuzzer corpus output) remains separate and is not affected.
