## Verdict

**UPDATES_NEEDED**

The C fuzzer plugin implementation (commits 19cd599, f19bfab) adds significant new functionality that is not adequately documented in `CLAUDE.md` or `README.md`. The documentation currently describes the fuzzer as "Python MVP" and omits critical environment variables, build commands, and architectural changes related to C fuzzing.

- [x] **Lines 154-167: Add C fuzzer limitations**
  Add a new Known Limitation entry (item 12 or renumber existing items):

  **12. C fuzzer harness validation — allowlist-based security model**
  - The C fuzzer compiles and executes arbitrary C code generated by the AI model inside a Podman container with full seccomp enforcement (`sandbox/seccomp-fuzz-c.json`).
  - Layer 1 (host-side) and Layer 2 (container-side) AST validation prohibit:
    - `system()`, `popen()`, `exec*()` calls
    - Network syscalls (`socket`, `connect`, `bind`, `listen`, `accept`)
    - File operations outside the build tmpfs (`open`, `creat` with absolute paths)
    - Non-standard library includes (only `stdlib.h`, `string.h`, `stdint.h`, etc. allowed)
  - Harness size capped at 64 KB.
  - **This is NOT a sandbox escape prevention mechanism** — it is a defense-in-depth measure to reduce prompt injection risk. The Podman container with seccomp is the primary security boundary.
  - See `_c_worker.py` lines 36-87 for validation constants and `sandbox/seccomp-fuzz-c.json` for syscall allowlist.

  **Rationale:** The C fuzzer introduces a fundamentally different security posture than the Python fuzzer (`_worker.py` uses `eval()` with restricted globals; `_c_worker.py` compiles arbitrary C with gcc). This should be documented prominently.

## Rationale

### Why These Changes Are Needed

1. **Commit 19cd599** ships a production-ready C fuzzer plugin with:
   - Entry point in `pyproject.toml` line 67
   - Dedicated container image (`sandbox/Containerfile.fuzz-c`)
   - Three new environment variables (`DCS_FUZZ_C_*`)
   - New Makefile targets (`build-fuzz-c-sandbox`, `test-c-fuzzer`)
   - Distinct security model (no `eval()`, gcc compilation instead)

2. **The architecture comment on line 32 of CLAUDE.md** still says "Python MVP", which is factually incorrect. The C plugin is fully integrated and tested (90%+ coverage per QA report).

3. **The environment variables table is incomplete**, omitting three variables that are parsed in `shared/config.py` and used by the C fuzzer. Users who need to configure C compilation flags or include paths have no documentation to reference.

4. **The Known Limitations section does not document the C fuzzer's security model**, which differs fundamentally from the Python fuzzer. This creates a risk that users will enable C fuzzing without understanding its attack surface.

5. **The README's "Supported Languages" section lists C for SAST only**, not mentioning that C fuzzing is also available. This is misleading for users evaluating the tool's capabilities.

6. **The `DCS_FUZZ_ALLOWED_PLUGINS` variable** is mentioned in both docs but neither explains how to enable multi-language fuzzing (e.g., `python,c`). The bridge resolver logs a warning if `c` is not in the allowlist (`bridge/resolver.py` lines 91-93), but users have no documentation pointing them to the solution.
